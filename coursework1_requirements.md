## 5. 交付物要求（Deliverable Requirements）

### 5.1 代码仓库（Code Repository）

- **公开的 GitHub 仓库**，具有可见的提交历史，展示一致的版本控制
- 包含清晰的 **README.md**：描述设置说明和项目概述
- 代码必须 **可运行**，并对应口头答辩中展示的版本
- **缺少版本控制、README.md 或代码不可运行 → 该部分判定为 Fail**

### 5.2 API 文档（API Documentation）

- 可以是生成的或手写的文档（如 Swagger UI、Postman、Markdown）
- **清楚描述**所有可用的端点（endpoints）、参数（parameters）和响应格式（response formats）
- 包含 **示例请求和预期响应**（JSON 或等效格式，允许使用其他格式但需在报告中说明）
- 记录 **认证过程** 和 **错误码**（如适用）
- 必须在 README.md 中引用，以 **PDF 文件** 形式提供（如通过 HTML 生成则转为 PDF）
- **缺少 API 文档 → 该部分判定为 Fail**

### 5.3 技术报告（Technical Report）

- 简洁的书面报告，概述关键设计和架构选择
- 要看到 **个人对技术栈的选择**
- **论证** 编程语言、框架和数据库的选择
- **反思** 挑战、测试方法和经验教训
- 包含 **局限性** 和未来改进/开发的潜在方向
- 包含 **Generative AI 声明** 及对其使用的深思熟虑的分析（如果使用了的话）
- **缺少技术报告或 GenAI 声明 → 该部分判定为 Fail**

### 5.4 演示幻灯片（Presentation Slides）

- PowerPoint 演示幻灯片，用于口头答辩中展示工作
- 幻灯片必须遵循良好演示的通用准则
- 参考：[https://library.leeds.ac.uk/info/1401/academic-skills/130/presentations-oral](https://library.leeds.ac.uk/info/1401/academic-skills/130/presentations-oral)

## 6. GenAI 使用规则

### 6.1 基本规则

- 这是一个 **Green Light Assessment**（绿灯评估）：**AI 是评估的组成部分，应当被使用**
- 可以使用 GenAI 工具（如 Secure Microsoft Copilot）
- **必须声明** 所使用的所有工具、用途，并附上导出的对话日志样例作为补充材料
- 使用 GenAI 进行 **调试、规划或校对** 是 **完全允许的**
- **未声明** 的 GenAI 使用构成 **学术不端行为**

### 6.2 GenAI 使用水平与成绩等级对应关系


| 成绩等级                     | GenAI 使用水平                           |
| ------------------------ | ------------------------------------ |
| **40–49 (Pass)**         | 以不系统和随意的方式使用生成式 AI                   |
| **50–59 (Satisfactory)** | 低水平使用 AI，如辅助编写或调试代码                  |
| **60–69 (Good)**         | 以方法论上合理的方式使用生成式 AI                   |
| **70–79 (Very Good)**    | 中等水平使用 GenAI，如理解技术和新概念               |
| **80–89 (Excellent)**    | 高水平使用 GenAI 辅助创造性思考和方案探索             |
| **90–100 (Outstanding)** | 创造性地应用生成式 AI，如探索高层替代方案、重新构想前沿解决方案的设计 |


### 6.3 使用 AI 处理数据集的推荐工作流程

1. **数据集发现**：使用 AI 搜索感兴趣领域的数据集
2. **下载与探索**：下载数据集并检查其结构、格式和内容
3. **生成导入脚本**：提供数据集子集给 AI，让其帮助创建导入脚本将数据存入 SQL 数据库

## 7. 评估任务（Assessment Tasks）

### 7.1 最低技术要求（Minimum Technical Requirements）— 达到 Pass (40+)

- 实现 **至少一个数据模型**，包含完整的 **CRUD**（创建、读取、更新、删除）功能，并关联到 **数据库**
  - 可以基于 **RESTful API** 设计
  - 也可以使用 **GraphQL API**（但必须在报告中解释选择 GraphQL 的理由）
  - 数据库：允许使用任何类型的 **SQL 数据库**；除非在报告中提供充分理由，否则 **不应使用 NoSQL 数据库**
- 包含 **至少 4 个 API 端点**，可通过 HTTP 请求访问
- 处理用户输入并返回适当的 **JSON 响应**（允许使用其他格式，需在报告中说明）
- 使用符合行业惯例的 **正确状态码/错误码**
- 可通过 **本地执行、Web 托管**（如 PythonAnywhere）或 **MCP 服务器** 进行演示

### 7.2 技术栈选择

- 可以基于 **Django** 实现
- 也可以使用其他合适的技术栈：如 **Python FastAPI**、**Node.js Express**、**Go Fiber** 等
- **必须在报告中论证技术选择和动机**

### 7.3 示例项目创意

- 图书元数据与推荐 API（集成公共数据集如 Goodreads 或 Google Books）
- 环境与城市气候统计 API（存储历史天气、空气质量和温度数据）
- 生产力与习惯分析 API（跟踪任务、习惯、完成模式）
- 公共交通可靠性与延误分析 API（使用开放英国交通数据集）
- 营养与食谱分析 API（基于开放食品数据集如 USDA 或 UK FSA）
- 房屋市场与租赁洞察 API（整合 ONS 或 Land Registry 数据）
- 文化活动与旅游洞察 API（使用博物馆、节日、遗产地的公共数据集）
- 体育表现与比赛统计 API（基于公开体育数据集如足球、板球或 F1）
